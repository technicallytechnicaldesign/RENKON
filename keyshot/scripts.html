Scripts

Automation for the Creo → KeyShot pipeline — batch views, turntables, and camera-driven reveal animations, written against the lux / luxmath scripting API. Headless-compliant where noted, so batch stages can run unattended on a licensed KeyShot install.

Inventory

0VAL
0_VAL_STEP_PREFLIGHT_AA01.py
STEP-file preflight — not a lux script, runs on plain STEP text before KeyShot ever opens the file. Checks the declared unit/scale (the #1 reported KeyShot import complaint), filename hygiene, duplicate base names, and best-effort material-name hints cross-referenced against a 1_HLP_MAT_LOOKUP_AA01.py-format mapping file. Forum-grounded: see RESEARCH_CREO_KEYSHOT.md.
Current
0VAL
0_VAL_IMPORT_HEALTH_AA01.py
Trial-import geometry/appearance health check — a fast, no-render gate before committing a folder to the full batch pass. Flags import failures, bounding-box anomalies, assembly-named files that imported as a single object (missing components), and parts with no Studios or cameras. Uses the now-confirmed getBoundingBox() call.
Current
0VAL
0_VAL_ORIENTATION_CHECK_AA01.py
Orientation / up-axis sanity check — computes a footprint ratio per part from its bounding box and flags statistical outliers relative to the rest of the batch. Explicitly a review-flagging heuristic, not a fixer or a ground-truth checker — a flagged part might be legitimately different, and an entire batch sharing the same bad orientation setting wouldn't be caught at all.
Current
2ABAT
2a_BAT_STD_VIEW_AA01.py
Studios-first multi-view batch render. Imports every model of a chosen extension from a folder, applies an optional material template, then renders every Studio (camera + environment pairing) defined per part — falls back to raw cameras only when a part has no Studios. Logs a CSV manifest.
Current
2ABAT
2a_BAT_TURNTABLE_AA01.py
Studios-first 360° turntable batch. Same Studio-aware camera/lighting selection as the std-view script; spins each imported part and encodes a small mp4 per part. Kept as a continuous spin on purpose — the "fade between angles" idea is planned as a separate 3_ post-process script instead of a change here.
Current
2BANI
2b_ANI_HERO_REVEAL_AA01.py
Automated hero-reveal shot: single monotonic zoom (in or out) into a camera/Studio's own framing, with optional vertical crane, HDRI rotation, brightness ramp, and synced Model Set animation playback. Orbit/arc was deliberately removed (Rodrigues' formula degenerates when the rotation axis aligns with the camera-to-object direction); ground-setter method name and getAnimationInfo()'s return shape are unconfirmed on your build — the script probes and reports which worked. See in-file docstring for full design notes.
Current
1HLP
1_HLP_MAT_PREFLIGHT_AA01.py
Material-template coverage QC, ahead of a full batch render. Imports every part in a folder, applies a material template, and reports which objects' materials did not change — a strong signal the template has no entry for that object's native material name. Does no rendering, so it runs in seconds.
Current
3PRC
3_PRC_CONTACT_SHEET_AA01.py
Standalone HTML contact sheet from a batch's rendered output (not a lux script — run with a regular Python 3 interpreter). Scans an output folder for stills and turntable videos, groups them by part, writes one contact_sheet.html.
Current
1HLP
1_HLP_MAT_LOOKUP_AA01.py
Creo → KeyShot material-name lookup table. Loads a CSV/JSON mapping of native material/appearance names to KeyShot material templates, then applies the matching template per object (not scene-wide) across a batch import. Prevention, complementing what 1_HLP_MAT_PREFLIGHT_AA01.py catches after the fact. Per-object template assignment is an unconfirmed API on this build — probes a short list of candidate setters and reports which worked.
Current
2BANI
2b_ANI_CUTAWAY_REVEAL_AA01.py
Cutaway / cross-section reveal. Sweeps a clip plane through an assembly's bounding box over a frame sequence with the camera held static — contrast to the hero-reveal script's camera dolly. Reuses the same Studios-vs-Cameras selection, manifest logging, and headless-dialog pattern as 2b_ANI_HERO_REVEAL_AA01.py. Clipping-plane and bounding-box APIs are unconfirmed on this build — probes several candidate method names and degrades to unclipped rendering with a warning if none resolve.
Current
2BANI
2b_ANI_ASSEMBLY_PROCEDURAL_AA01.py
Assembly reveal, procedural modes. Parts converge into their authored final positions while the camera holds static — four selectable modes: scatter_settle, staggered_build, spiral_converge, and ghost_fade. Motion is built from relative per-frame transform deltas that always sum to zero by the final frame — the only confirmed per-node mutator in this pipeline. Per-node transform composition, bounding-box sizing, and opacity control are all unconfirmed on this build; each is probed defensively and degrades gracefully. Sub-assembly build (grouped convergence) is a follow-on, not implemented here.
Current
3PRC
3_PRC_FADE_REEL_AA01.py
Multi-angle fade reel — a "turntable fade" built from fixed stills instead of a continuous 360° spin. Standalone (not a lux script); scans a batch's rendered stills, groups by part, and writes one self-contained HTML file per part that crossfades between its views on a pure-CSS @keyframes loop — no ffmpeg, no external dependencies.
Current
4CHK
4_CHK_AUDIT_AA01.py
Naming-compliance + render-completeness audit — the first script in stage 4. Validates every .py filename in a folder against the {PREFIX}_{AREA}_{NAME}_{REV}.py convention (skipping archive/), and cross-checks every *_manifest.csv row's output path against what's actually on disk, flagging missing files and failed/skipped statuses separately.

Conventions

  • Filename: {PREFIX}_{AREA}_{NAME}_{REV}.py
  • PREFIX: 0_ pre-check · 1_ pre-render/helper · 2a_ batch render · 2b_ animation · 3_ post-process · 4_ post-check
  • AREA: 3-letter code — BAT, ANI, HLP, PRC, NGS, CHK…
  • REV: XX01 — letter steps for a breaking change, number steps for a routine one
  • Python 3.10+, lux / luxmath modules
  • Headless CLI runs anything marked HEADLESS COMPLIANT; still needs a machine with a licensed KeyShot install
  • Studios (camera + environment + image style) preferred over raw Cameras
  • Binary scene files (.bip, .ksp…) tracked via Git LFS
